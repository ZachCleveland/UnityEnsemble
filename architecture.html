Project Overview

Top-down 2D game with 5 levels that begins with absolute silence and incrementally builds the theme song.

Each level requires solving different puzzles (box-pushing, memory, etc.) to receive access to an instrument.

Once the instrument is acquired, its musical component of the piece is added to the OST.

- Upon finishing all of the levels, all instruments serenade the user. Final ending of the game is still in planning stages.

Key Architectural Drivers

Because we are developing a puzzle game that has multiple levels, the ability to edit the puzzles is a high requirement, so that we can adjust difficulty and test for softlocks, etc.

We also need to easily distinguish between the different aspects of the game (UI, sound, coding, etc.) so we can easily tie them together.

Knowledge source and control components may produce new data objects that are added to the blackboard which is essential as player progresses the game and sequentially the OST

Architectural Style Choices

Blackboard

- Good for reusability

- Easy to update and modify

- Allows multiple processes to work closer together on separate threads

- Instantiatable as individual processes used in layer architecture.

Layered

- Simplicity of implementation

- Code organization is consistent

- Enables the team to work on different parts of the application with minimal dependencies

Our Architecture

Conclusion

Architecture chosen: Blackboard

- Reasons: For each level, we can easily keep track of the different states of the object inside of said level, such as keeping track of the locations of the boxes, whether or not an object has been acquired, or what instruments are playing at the time.

- Risks: possible problems in synchronization of multiple agents, high dependency on the knowledge source.

Our questions: Do we need individual blackboards or just one? Is there a more efficient way of storing this information?
